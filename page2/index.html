Merging Lanyon's stylesheets with your own

29 Oct 2014

mdo's Lanyon theme for Jekyll is pretty swell. It has a good design and is dead simple to use. I recently wanted to add a bit of custom styling and merge my new stylesheet with the existing ones. Here's how I did that.

Lanyon's existing style

Out of the box, Lanyon provides three different stylesheets:

  • poole.css
  • syntax.css
  • lanyon.css

These contain the different styles for headers, code blocks, sidebars and the likes, and are included in the _includes/head.html file in the following order:

<link rel="stylesheet" href="/public/css/poole.css">
<link rel="stylesheet" href="/public/css/syntax.css">
<link rel="stylesheet" href="/public/css/lanyon.css">

What I changed

What I wanted to do was merge all the files in one single aliconnors.css file (to reduce the number of http requests while being able to keep the original files).

Although the right way to go would be to use a css minifier, I intend to change my workflow soon to use Sass, so I didn't want to change things too much.

I ended up creating a CSS file which contains my custom rules and includes the original CSS using Front Matter:

---
layout: nil
---

{% include {{ site.baseurl }}public/css/poole.css %}
{% include {{ site.baseurl }}public/css/syntax.css %}
{% include {{ site.baseurl }}public/css/lanyon.css %}


// below are my custom rules

/*Center post images horizontally */
.post img{
    margin-left:auto;
    margin-right:auto;
}

/* Center .description and color them in dark grey */
.post .description{
    text-align:center;
    color:#9a9a9a;
}

Front Matter includes in my CSS

This merges all four files into one. After that, I simply updated my head.html to include the correct file.

<link rel="stylesheet" href="/public/css/aliconnors.css">

What's next?

I'll be doing bigger, more radical changes on my workflow soon. I'll try to post explanations for them here.

Bonus

I escaped the liquid tags in the code samples above using the raw and endraw tags, as described in this stackoverflow answer.

Modifying the Bootstrap dropdown on scroll

28 Oct 2014

I recently answered a Stackoverflow question about getting a dropdown menu to change the direction in which it opens depending on its position in the window. Here's a little explanation about how I got things to work.

Problem overview

The idea here is to get the Bootstrap dropdown component to have a more user friendly way of opening when it is displayed at the bottom of a page. If the dropdown is in the lower part of the page, it is a bit painful for the user to have to scroll down to see the menu items.

This is not cool, Bootstrap!

The person who asked the question wanted to be able to change the way the dropdown opens depending on its position in the window. If the dropdown wa below the vertical center of the window, it had to be opened upwards, otherwise downwards.

Final result

I made two iterations of the dropdown: the first changes dynamically when the user scrolls:

The magic happens when scrolling

The second - which I personally find less intrusive for the user - does not change states while it is open, but calculates it's most efficient state before opening:

The magic opens when opening the dropdown

Explanation

Both answers calculate a value that I named relativeOffset which is the offset between the top of the window - not the top of the page - and the dropdown toggle. By comparing this to the window height, we can know whether the dropdown is in the upper or lower part of what's visible in the window.

All we have to do is listen to the right event - $(window).scroll for the first solution, $('.dropdown-toggle').click for the second - and toggle the reverse style if we want to.

Here's the code for the reverse class:

.reverse{
    top:auto;
    bottom:100%;
}

And here's the javascript for the second iteration:

$(".dropdown-toggle").click(function(){
    // get the scollTop (distance scrolled from top)
    var scrollTop = $(window).scrollTop();
    // get the top offset of the dropdown (distance from top of the page)
    var topOffset = $(".dropdown").offset().top;
    // calculate the dropdown offset relative to window position
    var relativeOffset = topOffset-scrollTop;
    // get the window height
    var windowHeight = $(window).height();

    // if the relative offset is greater than half the window height,
    // reverse the dropdown.
    if(relativeOffset > windowHeight/2){
        $(".dropdown-menu").addClass("reverse");
    }
    else{
        $(".dropdown-menu").removeClass("reverse");
    }
});

A little something about anchors without href

28 Oct 2014

Today, I spent quite some time wondering why an anchor element on my page didn't have the correct cursor property. This quick post explains why.

The right way to use a

All web developers have already used anchor (a) elements without specifying the value of the href attribute. Often, we want to have a link that opens a modal box or fires an ajax call. We want it to look like a link but not to trigger a redirection.

Historically, this was possible using either one of the following methods:

<a href="#">Click me I'm a link</a>

<a href="javascript:void(0)">Click me I'm a link</a>

The difference between these two methods being that the latter will not scroll the page whereas the first method will bring you back to the top. See this fiddle for an example.

The HTML5 specification changed that by making the href attribute optional. You may now create placeholder hyperlinks that will be links, but will not trigger a redirection. And you can still do fancy stuff with them if you want to.

There is a slight problem

If there weren't any problems, things would be too perfect to be true and us lowly devs would end up being useless.

If you create a placehoder link in your website, the cursor property will not always be set to pointer as it should. As a matter of fact some browsers will style that as a simple text element. See this fiddle for an example.

In the end, you're better off using the good old javascript:void(0) method!
